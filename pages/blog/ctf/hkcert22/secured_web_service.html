Thought Process:


When checking the website, it immediately redirects the user to this page: [Ghost Astley Tokyo]

At first, we thought it may be related to the website itself, so we decided to use Linux to wget the website's source code.


However, as shown above, the source code of the website is not really that helpful in solving the puzzle.



nginx.conf file


We soon started to find documentations about the nginx.conf file.

We suspected that line 42-47 was where the vulnerability was located, as it was related to the flag.

Therefore, we looked up documentation about the alias path: https://nginx.org/en/docs/http/ngx_http_core_module.html#alias

We realized that there should be a slash at the back of /flag, which was missing in nginx.conf. 

location /flag(/ missing here) {
    # First attempt to serve request as file, then
    # as directory, then fall back to index.php
    #try_files $uri $uri/ /index.php?q=$uri&$args;
    alias /var/www/html/flag/;
}

After doing some research, with the keywords: 'nginx path traversal vulnerability' (that was not what we searched during the CTF, but we still managed to find similar results online about the exploit of doing ..), we found out that path traversal is possible if the / is removed.


https://www.acunetix.com/vulnerabilities/web/path-traversal-via-misconfigured-nginx-alias/





Solution


Use this knowledge to traverse the link (Tip: .. means go backwards one directory.)
